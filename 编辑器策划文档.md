# 博客编辑器策划文档

## 1. 项目目标

为当前 `Gmeek + GitHub Pages` 博客增加一个仅依赖静态托管的编辑器页面，提升写作体验，同时保持现有基于 Issue 的发布流程不变。

## 2. 当前限制

- 只有一个 GitHub 账号
- 站点仅部署在 GitHub Pages
- 没有独立服务器，不能运行真正后端
- 不能在前端硬编码高权限 Token

## 3. 方案定位

采用“方案 3 + 开源编辑器集成”：

- 在博客项目中新增 `docs/admin.html`
- 编辑器内核升级为开源项目 `Milkdown`
- 编辑器负责新文章内容编写、预览、片段插入、草稿保存
- 发布时跳转到 GitHub 新建 Issue 页面
- 由用户手动提交 Issue
- 之后交给 GitHub Actions + Gmeek 生成博客页面
- 保持 Markdown 作为唯一发布真源，兼容 GitHub Issue / GFM
- 通过 `markmap` 代码块约定为后续思维导图前端渲染做准备

## 4. 编辑器选型

本项目当前升级选择 `Milkdown`，原因如下：

- 纯前端可运行，适合 `GitHub Pages`
- 更接近现代富文本编辑体验，但底层仍是 Markdown 驱动
- 更适合“GitHub Markdown Issue 语法 + 富文本交互”的组合路线
- 后续更容易扩展 slash command、快捷插入、脑图模板等能力
- 仍然可以保持最终输出为 GitHub Flavored Markdown

### 4.1 选型补充说明

- `Vditor` 依旧是可用备选，但当前版本改为 `Milkdown` 作为主要编辑器
- 当前阶段不直接使用 `Tiptap`，是因为它更偏富文本优先，而本项目必须优先兼容 GitHub Issue Markdown
- 当前阶段不引入重量级思维导图编辑器，而是先采用“Markdown 约定 + 前端渲染”的方式降低复杂度

## 5. 本次已实现功能

- 标题输入
- 标签输入与标签快捷选择
- Markdown 富文本写作体验
- 本地草稿保存与清空
- 一键打开 GitHub 新建 Issue
- 集成 `Milkdown` 开源编辑器
- 支持 Markdown 驱动的富文本编辑体验
- B 站视频嵌入片段插入
- 网易云音乐嵌入片段插入
- 浏览器本地草稿保存
- 标题 / 引用 / 代码块快捷插入
- 思维导图 `markmap` 模板快捷插入

## 6. 嵌入内容策略

### B 站视频

通过插入 Gmeek 可解析的 HTML iframe 片段实现：

```html
Gmeek-html<iframe src="https://player.bilibili.com/player.html?bvid=BVxxxxxxxxxx&page=1" scrolling="no" border="0" frameborder="no" framespacing="0" allowfullscreen="true" width="100%" height="460"></iframe>
```

### 网易云音乐

通过插入 Gmeek 可解析的 HTML iframe 片段实现：

```html
Gmeek-html<iframe frameborder="no" border="0" marginwidth="0" marginheight="0" width="100%" height="86" src="https://music.163.com/outchain/player?type=2&id=歌曲ID&auto=0&height=66"></iframe>
```

### 思维导图

思维导图内容当前采用 `markmap` 代码块约定：

~~~md
```markmap
# 思维导图主题
## 一级节点
### 二级节点
## 另一个一级节点
```
~~~

这一约定的优势：

- 在 GitHub Issue 中仍然是合法 Markdown 代码块
- 不破坏当前 `Gmeek + GitHub Issue` 发文流程
- 后续可在博客前端扫描 `markmap` 代码块并自动渲染为思维导图
- 便于后续加入“实时脑图预览”能力

## 7. 用户使用流程

1. 打开 `admin.html`
2. 输入标题、标签、正文
3. 使用快捷按钮插入标题、引用、代码块等模板
4. 如需视频或音乐，在嵌入输入框填写 BV 号或网易云歌曲 ID
5. 如需思维导图，插入 `markmap` 模板并补充节点层级
6. 点击“新建 Issue 发布”
7. 跳转 GitHub 后检查内容并提交 Issue
8. 等待 GitHub Actions 自动生成博客页面
9. 如需修改旧文章，直接到 GitHub 对应 Issue 中编辑

## 8. 安全边界

- 不在前端保存 GitHub Token
- 不直接调用 GitHub 写接口
- 所有写操作仍由 GitHub 官方页面完成
- 草稿只保存在当前浏览器的 `localStorage`
- 当前页面默认定位是“新文章编辑器”，不是完整文章管理后台

## 9. 后续可迭代方向

- 增加文章封面快捷插入
- 增加常用模板库
- 增加一键生成摘要
- 增加文章大纲导航
- 增加本地导入/导出 Markdown 文件
- 增加编辑旧文章时自动读取 Issue 内容的辅助说明页
- 增加“发布检查清单”，避免忘记标签或标题
- 增加 `markmap` 思维导图实时预览面板
- 在文章页增加 `markmap` 代码块自动渲染
- 增加 Markdown / 富文本双视图切换

## 10. 适合当前项目的其他备选

- `Vditor`：原方案可继续维护，但当前不再作为主编辑器
- `Toast UI Editor`：成熟稳定，也适合静态接入
- `EasyMDE`：更轻量，但成品感弱于 `Milkdown`
- `Editor.md`：老牌可用，但样式相对旧
- `Tiptap`：富文本能力强，但与 GitHub Markdown Issue 的双向同步成本更高

## 11. 结论

在无服务器、仅 GitHub Pages 的约束下，这个方案仍然兼顾安全性、可维护性和写作体验。升级到 `Milkdown` 后，前端编辑体验更强，同时保持 Markdown 作为 GitHub Issue 的最终内容格式；配合 `markmap` 代码块约定，也为后续“思维导图前端展示”留出了清晰升级路径。
